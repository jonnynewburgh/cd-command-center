## 3. Latent-bug sweep

- **BOOLEAN-vs-INTEGER:** none. No boolean columns in either table.
- **Reserved-word collisions:** checked — `tax_year`, `total_revenue`,
  `net_income`, `notes_payable`, etc. are all non-reserved in both backends.
  No `"year"`, `"total"`, `"grant"`, or other collision-prone bare names.
- **Schema shape:** flat (one row per EIN in `irs_990`, one row per
  `(ein, tax_year)` in `irs_990_history`). NOT per-schedule. No Schedule A / B / C
  / I / J-detail / Part IX-B tables. Everything lives in the two flat tables.
- **Types:** Postgres `real` ≈ float4 vs SQLite `REAL` = 8-byte float
  — same precision artifact we've logged previously (e.g. MEDCURA HEALTH
  `program_service_revenue` is `60119256.0` in PG vs `60119255.0` in SQLite,
  ~$1 drift). Pre-existing, not caused by this fix.
- **`ON CONFLICT(ein) DO UPDATE ... updated_at=CURRENT_TIMESTAMP`:** valid on
  both backends after `_adapt_sql` wrap (both accept `CURRENT_TIMESTAMP`).
- **`upsert_990_history` has no `updated_at`:** intentional — history table's
  schema omits it (see db.py:421-438).

## 4. Scope classification

### What the 990 tables actually contain

**Dataset:** 990 data for **public-charity (501(c)(3)) nonprofits only**. NOT
990-PF. NOT 990-EZ. NOT 990-N.

**Universe:** Charter school LEAs (NTEE B29 dominant — 1,078 rows),
FQHCs (NTEE E32 / E320 / E32Z — ~400 rows), and CDEs. GA has 66 rows.

**Schema coverage (parts of the 990 form represented):**

| 990 part | Coverage |
|---|---|
| Part I (Summary) | Partial — `total_revenue`, `total_expenses`, `net_income`, `officer_compensation` |
| Part VIII (Revenue detail) | `program_service_revenue` only |
| Part IX (Functional expense detail) | `program_service_expenses` only |
| Part X (Balance sheet) | `total_assets`, `total_liabilities`, `unrestricted_net_assets`, `cash_savings`, `accounts_payable`, `accrued_expenses`, `notes_payable` |
| Schedule A (public charity status) | **Not stored** |
| Schedule B (contributors) | **Not stored** |
| Schedule C (political/lobbying) | **Not stored** |
| Schedule I (grants) | **Not stored** |
| Schedule J (officer comp detail) | **Not stored** — only flat `officer_compensation` scalar |
| Schedule L, M, O, R | **Not stored** |

**Data sources by row (PG):**

```
data_source='IRS'       3450 rows   (IRS e-file 990 XML via etl/fetch_990_irs.py)
data_source='ProPublica' 640 rows   (ProPublica API via etl/fetch_990_data.py)
data_source='IRS BMF'     33 rows   (BMF stubs — names only, no financials)
```

### PRI (Program-Related Investment) relevance — KNOWN PIPELINE GAP

User explicitly asked about PRI / Part IX-B coverage.

**PRIs are reported on Form 990-PF (private foundations), not Form 990
(public charities).** The specific field is **Form 990-PF Part IX-B Line 1a–1g**,
which itemizes PRIs with investment type, purpose, and amount.

The `irs_990` / `irs_990_history` tables in this codebase:

- Contain **zero 990-PF filers** (the universe is charter schools + FQHCs +
  CDEs, all public charities).
- Contain **zero Part IX-B columns** (no PRI-type, PRI-purpose, PRI-amount,
  or any PRI-related field).
- Contain **zero description/narrative text fields** — the schema is purely
  scalar financial totals. There is no freetext column where PRI descriptions
  could live as a fallback.

**Implication:** fixing the Postgres-compat bugs on `irs_990` /
`irs_990_history` does NOT, in itself, enable PRI data capture. PRI ingestion
needs a separate workstream:

1. New filer universe — discover GA-headquartered 990-PF filers (private
   foundations). Distinct source list from schools/FQHCs; can pull from
   the IRS BMF by NTEE `T20`/`T21`/`T22` in GA.
2. New data source — IRS 990-PF XML has a dedicated Part IX-B section
   (`ProgramRelatedInvestments` element tree). Separate XPath extraction
   from what `etl/fetch_990_irs.py` does today.
3. New schema — one row per `(ein, tax_year, pri_line)` with investment
   type, purpose text, and amount. Not a natural fit for the existing flat
   `irs_990` table.
4. Separate ETL script — `etl/fetch_990pf_pri.py` or similar.

**For this sweep:** the 990-table Postgres fix is a prerequisite (the rest
of the PRI pipeline design can't be validated while the current upserts
can't even run on PG), but it does not itself add PRI capability. This is
the expected "known pipeline gap" result parallel to the ECE (CO-only,
no operator intel) and FQHC (no UDS) diagnoses.

## 6. Proposed edits

**Scope: four one-line fixes in db.py, all identical pattern to
c87742b / 0b63cd7 / 8a8d848 — wrap `cur.execute("… ?", params)` with
`_adapt_sql(…)`. No schema changes, no logic changes, no refactors.**

### 6.1 `upsert_990` (db.py:2470)

```python
# before
    cur.execute(
        f"INSERT INTO irs_990 ({','.join(columns)}) VALUES ({placeholders}) "
        f"ON CONFLICT(ein) DO UPDATE SET {update_clause}, updated_at=CURRENT_TIMESTAMP",
        values,
    )

# after
    cur.execute(
        _adapt_sql(
            f"INSERT INTO irs_990 ({','.join(columns)}) VALUES ({placeholders}) "
            f"ON CONFLICT(ein) DO UPDATE SET {update_clause}, updated_at=CURRENT_TIMESTAMP"
        ),
        values,
    )
```

### 6.2 `upsert_990_history` (db.py:2678)

# after
    cur.execute(
        _adapt_sql(

## 8. Known pipeline gap — PRI / 990-PF (for separate workstream)

Summarizing from §4:

1. `irs_990` / `irs_990_history` contain only **public-charity 990** data.
   No private foundation 990-PF filings. No Part IX-B (PRI detail).
2. The PRI pipeline requires a separate filer universe (GA 990-PF filers,
   `T2*` NTEE codes), a separate data source (990-PF XML `ProgramRelatedInvestments`
   element tree), a separate schema (granular per-PRI rows), and a separate
   ETL script. Estimating: ~1 day of work analogous to the existing
   `fetch_990_irs.py` but for 990-PF.
3. Fixing Postgres compat on `irs_990` is a **prerequisite** for the PRI
   work (shared `db.py` / shared `_adapt_sql` path), but not a **component**
   of it.

Flagged so this does not creep into the current PR.
