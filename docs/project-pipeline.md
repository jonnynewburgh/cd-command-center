# CD Command Center — Project Pipeline

Living document tracking outstanding workstreams. Organized by 
category, not priority. Priority is decided per session based on 
current needs, not enshrined in the doc.

**Maintenance rules:**
- When an item ships, remove it from this doc in the same commit 
  that ships it. The commit message carries the history.
- When a new workstream is identified, add it to the appropriate 
  category with the same level of detail as existing items.
- When a scope question is unresolved, name it explicitly in the 
  item (see D1 for the pattern).
- Do not add items for bugs that should just be fixed — this is for 
  workstreams, not a bug tracker.
- Do not let categorization become a project of its own. If an item 
  doesn't fit cleanly, put it in the category it's closest to and 
  move on.

B6. cur.lastrowid and dict(row) on Postgres. save_user_note (db.py:3071) and save_document (db.py:3170) execute bare INSERTs and then return cur.lastrowid to FastAPI callers. api/routers/notes.py:48 returns that id to the dashboard as {"note_id": ...}. psycopg2's cursor.lastrowid returns None for INSERT statements (Postgres tables have no OID by default), so both endpoints silently return {"note_id": null} on Postgres. Related sibling bug: get_bookmarks (db.py:3108) and get_user_notes list-building use [dict(row) for row in cur.fetchall()] — psycopg2's default cursor returns plain tuples, not sqlite3.Row objects, so dict(row) raises TypeError which the surrounding silent except swallows, returning []. Same bug class: SQLite-specific row/cursor conventions that fail differently on psycopg2's default cursor. Fix approach: either switch get_connection() to psycopg2.extras.RealDictCursor (large blast radius — affects every SELECT path in db.py; changes row access from tuple-index to dict-key everywhere) or add a small row_to_dict(cur, row) helper + use RETURNING id for INSERT paths that need lastrowid. Decision needed before implementation. Flagged by the B5 inventory (2026-04-22); standalone workstream. Surfaced again by the save_bookmark verification (get_bookmarks on Postgres returned [] post-fix, proving this is a separate unblocked issue). **Per `docs/debug/postgres_readiness_inventory_2026-04-22.md`: cur.lastrowid is 3 sites total — adds L1 `db.py:1292 log_load_start` (NEW finding: pipeline run logging is silently broken on Postgres because run_id = None means log_load_finish UPDATEs zero rows; every pipeline shows status 'running' forever). dict(row) sweep is incomplete — at least 6 confirmed (`get_school_by_id`, `get_fqhc_by_id`, `get_ece_by_id`, `get_nmtc_project_by_id`, `get_user_notes`, `get_bookmarks`) plus 8 more in the latent tearsheet path; needs a fresh full-repo grep.**

Category C — New data ingestion (expand existing pipelines)
Adding real operator-intelligence data to pipelines that currently have shallow or directory-only data. Each is a full multi-hour session minimum; requires source research and schema design.
C1. FQHC UDS ingestion. Current fqhc table is site-directory-only (HRSA Health Center Service Delivery & Look-Alike Sites CSV — 11 identity columns, 3 organizational, 2 weak UDS-adjacent fields). Missing the underwriting-grade data: payer mix (Medicaid/Medicare/uninsured percentages), visit and encounter counts, clinical quality measures (diabetes A1c control, hypertension control, cancer screening rates), staffing FTEs by role, revenue/cost/grant-funding fields. HRSA publishes UDS annually with ~9-month reporting lag. New fqhc_uds_reports table keyed by BHCMIS org ID + year. Join key: BHCMIS org ID + year. Separate ingest pipeline; site directory remains the spine that UDS joins against.
C2. ECE operator intelligence. Current ECE table is licensing-directory-only — center identity, address, licensed capacity, licensing status. Missing: actual enrollment (vs licensed capacity), subsidy participation (CCDBG, state voucher programs, Head Start partnership, Pre-K funding source), quality ratings (state QRIS star ratings, NAEYC accreditation, CLASS scores), staffing ratios, child demographics, financials. Unlike FQHC where UDS provides a national source, no single national ECE operator-intelligence source exists — QRIS, subsidies, Head Start, and financials all come from different state and federal systems. Multi-source, state-by-state project. Substantially larger scope than C1.

Category D — New pipelines (entirely new data domains)
Building things that don't exist today. Biggest, most open-ended workstreams.
D1. Form 990-PF ingestion for PRI. Current irs_990 table holds Form 990 filings from public charities only (charter LEAs, FQHCs, CDEs). Private foundations file Form 990-PF, which includes Part IX-B (Program-Related Investments). PRI work requires a separate pipeline: pull IRS 990-PF bulk XML from AWS S3, parse Part IX-B, store raw description text (no LLM parsing, per prior hallucination issues). Different filer universe, different XML schema, different table.
Scope decision needed: Original scope was GA-headquartered foundations only (~500-1,000 filers) as a prerequisite for the GSIC Georgia market mapping deliverable. You've expressed interest in expanding to US scope (~100,000 filers) — that's a fundamentally different-sized pipeline (100-200x more data, different design constraints around storage and incremental ingestion). Before starting, decide explicitly: is GSIC still the primary customer with its timeline, or is this a longer open-ended ambition? They're different projects. Don't start until that's clear.
D2. 990 asset-class expansion. Current 990 coverage maps to existing asset-class pipelines (charter schools, FQHCs, CDEs). As the command center extends to new asset classes — supportive housing, workforce development, senior services, community health worker orgs, arts/cultural institutions, charter management organizations — the 990 ingestion needs to broaden. Each new asset class requires: (1) an EIN source list (IRS Exempt Organizations Business Master File, NTEE code filter, sector-specific directories), (2) a corresponding facility/entity table, (3) a join from irs_990.ein to that table's EIN column. Not a single project — an ongoing expansion done per asset class as underwriting demand drives it.
D3. Historic Tax Credit (HTC) integration. No current pipeline supports HTC underwriting. Adding it means: NPS/SHPO certification data (Part 1, 2, 3 applications), qualified census tract overlay for HTC-boosted projects, rehabilitation expenditure tracking, HTC/NMTC twinning analysis (common for deeply-distressed historic redevelopment deals). Scope before first deal that needs it — retrofitting underwriting data into an in-flight deal is more painful than having it ready.
